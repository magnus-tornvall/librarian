Component · per agent

Instrumentation

one thin adapter per agentan integration is just an event emitter

Role

Map native agent events to the canonical event schema, stamp Resource facts, emit cheap non-authoritative salience hints, append via the collector. Zero domain logic — adding a new agent (Claude Code, OpenCode, Codex) means writing a thin adapter, nothing more.

What it stamps

  • Facts, not identity: agent, agent_version, machine_id, cwd, git_root, git_remote, git_branch.
  • Authoritative project_slug is not on events — derived deterministically at distill/index time and cached in note scope. (Fixes SuperBrain's late-slug bug class: silent degradation to global-only recall.)
  • Hints: { possibly_salient?, reason? } — non-authoritative. Canonical salience lives in the collector/distiller.

Prior working assets

  • OpenCode plugin writing SuperBrain-format NDJSON, verified end-to-end — to be remapped to the canonical schema (roadmap step 6).
  • Fully researched OpenCode recall-injection design (used on the push side, see Push adapters).

Component · library + CLI

Collector

no daemonowns the prompt renderer

Write pipeline

native event → normalize → redact → validate → append
Redaction before durable append is non-retrofittable: secrets in an append-only replayable log are immortal. Redaction preserves correlation without the secret — [REDACTED:token:sha256:abc123] — and applies to prompts as well as commands.

Owns

  • Distill triggering — lazy, detached-child model: checkpoint acquires lockfile, spawns detached distill child, lock token via env, child releases. Stale-lock recovery via PID/token checks + timeout.
  • Authoritative salience and the distill-skip heuristic (0 salience markers, 0 write tools, <2 prompts, <10 events → skip).
  • Mechanical provenance stamping: the renderer presents events with ordinal indexes, the LLM cites indexes, the collector maps indexes → ULIDs. Provenance is never LLM-authored.
  • The prompt renderer — all LLM-facing serialization (see Note log → rendering, and Recall).

Guards

  • Hard-rejects record_class: diagnostic at the validate stage (quarantine-with-error, not silent skip).
  • Poison records quarantined with debug context; quarantine verdicts go to the diagnostics log. Partial trailing JSON lines ignored until completed.

Renderer (storage vs. prompt — the category rule)

Storage formats optimize for machines, humans, and replay; prompt formats optimize for signal-per-token. They meet only at the renderer, which carries zero compatibility obligations — prompts are ephemeral, never parsed back. Real token win = field elision (drop ULIDs, machine_id, schema_version; collapse timestamps; elide repeated cwd):

[1] 12:04 prompt "fix the login redirect bug, it loops on expired tokens"
[2] 12:05 write src/auth/session.ts
[4] 12:09 bash: git commit -m "fix: expire check before redirect"  ← salient:vcs_commit

Log · domain

Event log

sacred · append-only · replayable · never deleted~/.librarian/dataNDJSON per session

Role

Canonical, normalized, redacted-before-append telemetry from instrumentations and integrations. Source of truth for raw activity — the thing the distiller consumes and the thing provenance drill-down recovers verbatim excerpts from.

Schema (v1, draft → schema/event.md)

type CanonicalEvent = PromptEvent | ToolEvent | SessionEvent;
// ContentEvent reserved for non-agent sources — rule settled, shape deferred

EventBase = {
  schema_version: 1;
  event_id: string;          // ULID, generated before append
  ts: string;                // ISO 8601
  resource: { agent; agent_version?; machine_id; cwd;
              git_root?; git_remote?; git_branch? };
  context:  { session_id; turn?; cwd };
  hints?:   { possibly_salient?; reason? };  // non-authoritative
}
PromptEvent  = { type:"prompt"; prompt }              // redacted
ToolEvent    = { type:"tool"; tool:{native_name; canonical_name; category};
                 command?; files? }                   // command redacted
SessionEvent = { type:"session"; action:"start"|"stop"|"compact"|"checkpoint" }

Consumption

  • Cursors: {consumer, log_name, file_path, byte_offset, last_record_id?, updated_at} — advance only after successful processing.
  • machine_id: generated once, persisted at ~/.librarian/machine-id — never the hostname.
  • V1 is single-machine by declaration; ULIDs keep records collision-safe; per-machine segments are the eventual multi-machine answer (deferred, trigger: second machine).

Golden examples (acceptance test for schema concreteness)

(1) prompt in a git repo · (2) file edit → file_write · (3) git commit via bash → vcs_commit · (4) redacted command containing a token · (5) session checkpoint.

Components · the narrow waist

Distillers

distiller: llmdistiller: human
Nothing enters the note log without a distiller's judgment. There are exactly two distillers. No generic import path; machine-produced content — however pre-condensed — passes through the LLM distiller. The distiller is admission control, not compression: one writer discipline, one quality gate, one narrow waist. endorsed by literature round

LLM distiller

  • Consumer of the event log via an inference provider (claude -p pure completion first; provider swappable down to local).
  • Prompt selection keyed on origin — per-origin profiles, default profile for unconfigured origins. Mechanism deferred until the first non-agent source.
  • V1 revision rule: may revise only deterministic-ID notes (project:{slug}:summary, person:{name}, daily:{date}, curated:{id}), fetched only by deterministic ID — never search-and-mutate "probably related" notes. Everything else is episodic and immutable forever.
  • Re-distill invariant: idempotency is by provenance, not content — re-distilling an already-provenanced event range is a bug. Cursor advance-after-success + this invariant = exactly-once-ish. Nasty-path fixture required.
  • Never emits tombstones in v1 (CLI/human actions only).

Human distiller (curated-note importer)

  • Converts human-authored Markdown in vault/curated/** into note records directly — the human distiller's serialization mechanism, not an import facility.
  • Preserves the Markdown body verbatim in body.details; derives title from H1 and summary from the first paragraph. No LLM normalization — a correctness feature, not just fidelity.
  • Curated frontmatter may declare an explicit note_id (path-hash fallback); renames detected via content_hash tombstone the orphaned old ID.
  • Diagnostic insights enter memory only through this door — the raw traces never do.

Shared guards

  • Both hard-reject record_class: diagnostic.
  • Generated files are structurally excluded from curated ingestion (directory split). No mixed-ownership regions inside one Markdown file. Ever.

Quality gates (fixtures, not frameworks)

Provider qualification: 3–5 synthetic sessions in → note lands, routes correctly, links sanely — guards quiet degradation on small local models. Origin qualification: every new integration ships 3–5 golden content-events + expected outcomes.

Log · domain

Note log

sacred · append-only~/.librarian/datanotes/{yyyy-mm}.ndjson
The contract: everything in the note log was judged worth remembering, by a named judge (distiller), from a named origin (origin). The vault, the index, and recall are downstream conveniences of that one guarantee.

Identity & revisions

  • note_id = stable logical identity; revision_id = immutable version (ULID). Snapshot revisions, latest-revision-wins. No event-sourced note internals, no diff/patch format.
  • Identity modes: deterministic (revisable, keyed IDs) vs episodic ({type}:{ulid}, one revision, immutable). Semantic consolidation deferred — it is the entity-resolution problem.
  • Tombstones exist in the schema from day one; v1 emitters are CLI/human only.

Schema (v1, draft → schema/note.md)

NoteRevision = {
  kind:"note_revision"; schema_version:1;
  note_id; revision_id; previous_revision_id?;
  created_at;
  identity: { mode:"deterministic"|"episodic"; key? };
  source:   { origin;            // open vocabulary, MANDATORY, fail-closed
              distiller:"llm"|"human";
              model?; agent?; source_path?; content_hash? };
  note_type:"fact"|"decision"|"project_summary"|"person"
           |"daily"|"episode"|"curated";
  title;
  scope:      { project_slug?; git_root?; git_remote?; global? };
  provenance: { session_id?; event_ids?; event_range? };
  links:      Array<{ target_type; target; relation? }>;
  body:       { summary; bullets?; details? };
}
NoteTombstone = { kind:"note_tombstone"; …; source:{kind:"human"|"cli"} }

Source identity — two independent dimensions

origin (where: opencode, claude-code, human, email, … — open string vocabulary, denormalized at distill time, indexed) and distiller (who judged it in: llm | human). Write side and read side share no contract: a source may exist purely to enrich memory with no injection adapter, and vice versa.

Storage format (settled)

NDJSON stays for all logs — self-describing, nested-structure-capable, robust to payloads full of pipes/newlines, jq/SQLite-JSON tooling, glass-box ownership. Token efficiency is a renderer property, not a storage property; the LLM never reads a log file verbatim.

Component + store

Indexer & Index

SQLite FTS5 · BM25the one blessed index

Indexer

  • Independent note-log consumer with its own cursor.
  • search_text is indexer-derived by a fixed concatenation rule over title / summary / bullets / details / scope / links — never a record field.
  • origin indexed as a filterable column.
  • Fail-closed: records missing origin are excluded from the index.

Index decisions

  • BM25-over-SQLite-FTS5 only. No recall-provider abstraction. Schema must not block later vector search — but nothing more. endorsed — lexical retrieval also has a milder distractor profile than dense retrieval
  • Recency decay exp(-ageDays/90) computed in code, half-life tunable — freshness/conflict resolution must not be LLM-reasoned.
First search upgrade is trigger-gated, not a roadmap item: if negative fixtures show distractor injections despite the relevance floor → add a re-ranking pass. Re-ranking before vector search; vector/hybrid only if BM25 + re-ranking proves insufficient.

Component · domain core

Recall

recall precision is the product surfaceread-only DB access

Scoring — deterministic, in code

BM25 → RRF-style fusion (where applicable)
     → project/global boost
     → recency decay exp(-ageDays/90)
     → weights: score × f(origin) × f(note_type)
       e.g. { human:1.5, opencode:1.0, email:0.6 }
            × { curated:1.4, decision:1.2, project_summary:1.0,
                fact:0.9, daily:0.7, episode:0.7 }
     → relevance floor

Authority ordering (reflected in weights, wording, and tests)

current workspace evidence
> explicit user instruction in current session
> curated memory
> deterministic project notes
> recent episodic notes
> old episodic notes

The first line is the poisoning guard: memory is always framed as subordinate to what the agent can see in the repo right now.

Every injection leaves a trace

Recall writes an injection trace (with injection_id, candidates, cut reasons, and a config snapshot/hash of the weights and floor in force) to the diagnostics log. Recall telemetry — injected count and tokens, query + top-result metadata, corrections/contradictions, turns-to-goal — is computed from those traces; no second pipeline. Measured from the walking skeleton onward.

Under invisible injection, wrong context is worse than no context. A single near-miss distractor measurably hurts. Fail-closed is extended from scope to relevance.

Contract · push path

Push adapters — invisible injection

opencode (two-phase hook)claude-code (parity)precision-first

The contract

  • 0–5 records per prompt, never force-filled. A low-relevance note filling a slot is a distractor; a distractor is worse than an empty slot. Relevance floor = BM25-score threshold, tuned against fixtures. endorsed — single-distractor harm is measured and monotonic
  • Budget ~300–700 tokens per prompt (SuperBrain's ~500 confirmed in-range). Do not grow the budget "to be safe."
  • Require project match or explicit global scope.
  • Layout rule: turn-1 brief + prefs ride the first user message and never mutate mid-session (stable prefix, cache-friendly); per-prompt recall splices adjacent to the latest user message (volatile suffix, pays full price by design). Top hit nearest the prompt.

Injected block shape — tagged, provenance-bearing, freshness-bearing, non-authoritative

<librarian-memory injection_id="01J…" indexed_through="notes/2026-07.ndjson:12345">
Possibly relevant prior context. Prefer current repository evidence and
current user instructions if they conflict.

1. [decision · curated · 2026-07-03 · high authority]
   The project favors KISS, file-over-app, minimal abstraction.
   src: curated:author-context

2. [project_summary · distilled · 2026-07-03 · medium authority]
   Librarian uses event logs as telemetry and note logs as canonical memory.
   src: session 01J…, events 01J…–01J…
</librarian-memory>

Two-phase hook: compute on chat.message, splice ephemeral tagged part on experimental.chat.messages.transform, open-thread pointers on experimental.session.compacting; avoid system.transform.

Contract · pull path

MCP server — model-initiated recall

the universal floorone implementation serves every MCP client
Different physics from push. The model chose to search: returning up to ~10 scored results with full metadata is appropriate. The austerity rules (0–5, floor, tight budget) are push-path rules — do not "fix" the MCP tool to obey them.

Tools

  • Search: scored results with full metadata; origin/scope filters exposed as tool parameters.
  • Provenance drill-down: given a note_id, fetch the verbatim event excerpts behind it via collector-stamped provenance. Distilled notes are a supplement to recoverable verbatim source, not a replacement. endorsed — supplement-not-replace recovers most verbatim performance at a fraction of the tokens Push stays notes-only; pull gets depth.

Why it matters

Pull is what delivers the cross-vendor promise: ChatGPT, Claude.ai, Gemini, and any MCP-speaking agent read the same memory. On-demand and model-initiated rather than invisible — the floor, not the premium experience. Precedent: SuperBrain's sb-mcp.

Components · note-log consumers

Exporters & the vault

Obsidian exporter firstSQLite mirror second

Exporter rules

  • Independent note-log consumers; idempotent by note_id; may filter by origin — per-source vault views are exporter configuration, nothing more.
  • One interface: exportNoteRevision(record). No plugin system.
  • Never render diagnostics into the vault, in any form — no debug-dashboard exporters.

The vault split (structural invariant)

  • vault/generated/** — exporter-owned, deterministic paths, librarian_generated: true frontmatter + <!-- librarian:generated; do not edit -->, overwritten freely.
  • vault/curated/** — human-owned, ingested by the human distiller, origin: "human".
Invariant: generated files are excluded from curated ingestion — enforced by construction (directory split), not policy. No mixed-ownership regions inside one Markdown file. Ever. This kills the generated-export feedback loop.

Obsidian remains the human-facing view, but is no longer canonical storage: durable memory records are the domain layer; Obsidian pages are a rendered view or a curated input.

Log family · self-observation

Diagnostics

~/.librarian/diagnosticsfreely deletable · never ingested · never in the vault

Purpose

Answer "why did this memory surface" — and "why not," and "why wasn't this session remembered." Explainability of deterministic decisions, not OTel-style request tracing. Same NDJSON/cursor/segment machinery as the domain logs; the opposite retention story.

What is logged

  • Injection trace per injection: injection_id (the same ULID threaded into the injected block), query, candidate note_ids with raw and post-weight scores, cut reasons (below_floor | budget | scope_mismatch), records shipped, indexed_through, and a config snapshot/hash of the weights/floor in force — day-one and non-retrofittable: traces without the policy-in-force become unexplainable the moment the config is tuned.
  • Distill verdicts: skip reasons, quarantine events, retry exhaustion.
  • Not logged: timings, latency spans, counters — retrofittable, currently useless. OTLP export deferred (trigger: wanting an external backend / something feels slow); a converter is a page of code.

Isolation — structural, enforced three ways

  • Placement: separate root, outside the data-log root and outside the vault. Deletable at any time with zero replay consequences.
  • Poison-pill: records carry record_class: "diagnostic" and deliberately do not conform to the canonical event shape. Every ingestion-side validator hard-rejects them — quarantine-with-error, not silent skip.
  • Fixture: feed a diagnostics file to the collector; assert loud rejection.
Rationale (recorded so the rule is understood, not just obeyed): self-observation entering memory creates a reflexive loop — the system forming memories about its own memory behavior, which influence recall, which generates new diagnostics. It is the generated-export feedback loop one level up, and it degrades quietly. Diagnostic insights may enter memory through exactly one door: a human writes a curated note. The raw traces never do.

Interface · glass-box on demand

CLI

more important than any daemon

Commands

  • librarian drain — process everything pending. The manual recovery and debug tool.
  • librarian why <injection_id> — replay the recorded injection trace.
  • librarian why-not <query> <note_id> — a pure function, no logging needed: replay the query against the current index, show where the note ranked and which gate cut it. Determinism makes the negative trace free.
  • librarian note show <note_id> --with-provenance — latest revision + source kind, session, event range, source excerpts.

Why glass-box matters

Vendor memory is invisible and uninspectable and uneditable — that is its weakness. Librarian is plain markdown and NDJSON you can read, correct, author, and veto. The mechanism behind the glass: provenance on every note, provenance and freshness in every injected block, an injection_id on every injection, and these commands.
